The prompt doesn't give guidance on search strategy\n\nDetection algorithm:\n```python\ndef detect_query_thrashing(tool_calls: list[ToolCall]) -> list[ThrashingEvent]:\n    search_calls = [c for c in tool_calls if c.tool in ('Bash', 'Grep', 'graph search')]\n    events = []\n    window = 5  # consecutive calls\n    for i in range(len(search_calls) - window):\n        cluster = search_calls[i:i+window]\n        queries = [normalize_query(c) for c in cluster]\n        if semantic_similarity(queries) > 0.7 and len(set(queries)) > 2:\n            events.append(ThrashingEvent(turns=cluster, theme=dominant_term(queries)))\n    return events\n```\n\nWhere `normalize_query` strips file paths and punctuation, and `semantic_similarity` can start as simple Jaccard similarity on word bags and upgrade to embeddings when we have volume.\n\n---\n\n## Part 3: Prompt Templating\n\n### 3.1 The Problem With Current Prompts\n\nThe sprint agent prompts are long and manually crafted. The harness can count tool calls mechanically, but it can't know whether a search was useful. The qualitative judgments — \"this tool was frustrating,\" \"this search failed\" — can only come from the agent's own model of what it was trying to do and what it got back.\n\nThe harness captures what happened; the experience report captures *how it went*.\n\n### D2: Structured Markdown vs. JSON\n\nExperience reports are structured markdown, not JSON. Reasons:\n- Agents are reliable markdown writers; JSON schemas add failure modes\n- Markdown is human-readable without tooling\n- Fields with free-form text (confidence rationale, decision justification) don't map cleanly to JSON values\n- Post-processing can parse markdown sections mechanically\n\nIf downstream tooling needs structured data, the parse step handles it.\n\n### D3: One Report Per Run, Not Per Tool Call\n\nGranularity is per-run. The experience report is a run-end synthesis, not a per-tool log. Per-tool call annotations would be noise and would interrupt the agent's flow. The synthesis at the end is where the agent's model of \"what worked\" and \"what didn't\" is most coherent.\n\n### D4: Honesty Incentive\n\nThe prompt for the experience report explicitly says: \"a polished report that hides real friction is less useful than a candid one that admits struggles.\" This is an anti-sycophancy instruction. Without it, agents will report all tools worked great and all searches were productive, which is the opposite of useful.\n\nTesting whether agents are honest: compare experience report confidence scores against Reviewer's independent assessment of the same session. If experience reports consistently claim 95% confidence when the Reviewer consistently grades the same sessions as partial, the honesty instruction needs strengthening.\n\n### D5: Threshold-Based Actuation, Not Automatic\n\nThe feedback loop produces *signals* and *beads*, not automatic prompt edits. Automatic prompt mutation based on agent reports is a fast path to prompt drift and instability. Humans review the aggregate signals, decide which ones warrant action, and make deliberate edits. The value is in making the signal visible, not in closing the loop without oversight.\n\nThis may relax over time as we build confidence in the signal quality. Start conservative.\n\n---\n\n## Appendix A: Existing Patterns Referenced\n\nThe following patterns from the jira project informed this design:\n\n| Pattern | Source | Influence |\n|---------|--------|-----------|\n| `session_state.txt` (IN_PROGRESS / COMPLETED / BLOCKED) | `bcf3b673-a4a` Sprint Agent Architecture | Completion status field |\n| `decision.txt` (CONTINUE / CONSOLIDATE / COMMIT / REMEDIATE) | `f1ee7002-2df` Sprint Orchestration Guide | Structured single-token decisions as control signals |\n| Session Summary (tool counts, turns, cost, duration) | `5924741c-265`, `4d14234c-3f7` harness outputs | Tool use metric baseline |\n| Process Evaluator (`test_eval.md`) + Result Evaluator (`user_eval.md`) | `06d1296a-1f3` JIRA Testing Harness | Dual evaluation streams |\n| KB-Ticket loop (gap detected → ticket → KB agent fills gap) | `06d1296a-1f3` JIRA Testing Harness | Self-expanding knowledge pattern |\n| Evidence-based prompt refinement (≤10% change per iteration) | `06d1296a-1f3` JIRA Testing Harness | Refinement discipline |\n| Reviewer as intelligence layer (session-NNN.md accumulation) | `bcf3b673-a4a` Sprint Agent Architecture | Session accumulation pattern |\n| Prompt = process, KB = knowledge (separation principle) | `06d1296a-1f3` JIRA Testing Harness | Shared blocks vs. agent-specific blocks |\n\n---\n\n## Appendix B: File Layout\n\n```\nagents/\n├── shared/\n│   ├── tool_guidelines.md          #
