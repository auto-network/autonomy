For **highly branching, partially overlapping document edits**, the state of the art is **not** “better Git diffs.” It is usually a **causal change DAG plus semantic merge structures**, most often **CRDT-based**, with the document state separated from the history/query layer. In practice, the strongest modern systems track edits as **operations with causal dependencies**, represent version states as **heads/frontiers/version vectors**, and compute diffs between arbitrary historical states instead of only doing parent/child diffs. That is the core model behind modern local-first systems such as Automerge, and similar ideas show up in Loro and patch-theoretic systems like Pijul.

The current frontier has roughly **three layers**:

1. **A causal history graph** of operations/changes.
2. **A semantic document model** for the thing being edited: plain text, rich text, tree, JSON, etc.
3. **Version-query machinery** that can answer “what changed between these two branch frontiers?” without replaying everything naively. Automerge explicitly exposes this model: documents have **change hashes**, can have **multiple heads**, and support working “at” historical heads rather than only at the latest state.

For **plain text**, the strongest trend is away from classic OT and toward **high-performance operation-based CRDTs** or close relatives. Yjs is still one of the most widely used production systems, especially for editor integrations. Automerge is the best-known “history-preserving, branch-aware” system, and Automerge 3 significantly reduced the traditional CRDT memory penalty by using its compressed columnar representation in memory as well as on disk. More recent work like **Eg-walker** pushes the frontier further on the algorithmic side by aiming for CRDT-style branching/offline merge semantics with much lower steady-state memory and faster loading/merging under long divergence.

For **rich text with overlapping marks**—bold/italic/comments/annotations crossing edits—the state of the art is **Peritext-style semantics**, not plain-text CRDTs with formatting bolted on. Peritext was specifically designed because ordinary text CRDTs do not preserve user intent well under overlapping formatting and concurrent insert/delete/format operations. Modern rich-text CRDT libraries increasingly adopt or emulate those ideas; Loro explicitly positions its rich-text model this way.

For **hierarchical document graphs**—outlines, block trees, folders, AST-like structures—the frontier is **tree CRDTs with native move semantics**. Historically, concurrent move was a disaster area; the important advance is work like **A highly-available move operation for replicated trees**, which gives a principled way to support concurrent moves while preserving tree validity. Loro’s movable-tree implementation is a concrete production-oriented example of this line of work.

If your emphasis is specifically **“highly branching yet overlapping edits”**, then the most interesting split is between:

- **Commit-oriented DAGs** like Git: excellent at explicit branches, weak at semantic overlapping edits.
- **Patch theory / commutation systems** like Pijul: branches are closer to **sets of patches**, and the model is much more natural when many branches share overlapping work in different combinations.
- **CRDT/history-graph systems** like Automerge/Loro: best when you want arbitrary offline edits, many heads, automatic merge, and branch/query over document history.

Pijul is still conceptually important here because it treats a branch as a **set of patches** rather than a privileged linear ancestry path, which is exactly the direction you want when overlap matters more than merge-base ancestry.

So if I had to summarize the **actual state of the art** in one sentence: it is **semantic, causally ordered operation graphs with compressed history storage and type-specific merge logic**, not line diffs. Automerge’s current architecture is probably the cleanest example of the “document as a change graph with first-class version heads” approach, while Peritext-style rich-text semantics and move-safe tree CRDTs are the key advances for nontrivial document structure.

What is **not solved** yet is also important. Even the best systems still struggle with:

- producing merges that match **human intent** under large, semantically conflicting concurrent rewrites,
- keeping metadata small enough for extremely long-lived docs,
- giving users understandable **review/track-changes** views over CRDT histories,
- and integrating semantic structures like tables, comments, embeds, citations, or AST nodes without bespoke rules. Eg-walker, Automerge 3, and newer rich/tree CRDT work are progress on performance and structure, but they do not make semantic conflict resolution disappear.

My practical take:

- For a **production system today**, the strongest default is **Automerge** if you care about full history, branching, offline sync, and arbitrary version queries.
- For **editor ecosystem maturity**, **Yjs** remains extremely compelling.
- For **rich text with overlapping annotations**, use a **Peritext-like** model or you will regret it.
- For **tree/block documents with moves**, require a **move-aware tree CRDT**.
- For **branch algebra / overlapping patch sets**, study **Pijul’s patch theory**, even if you do not literally adopt Pijul.

Then the problem stops being “document change tracking” and becomes **multi-layer causal knowledge management**.

The right mental model is:

- **Layer 1:** the editable artifact itself — text, blocks, rich text, tree, tables.
- **Layer 2:** a **structured knowledge layer** extracted from or embedded in that artifact — entities, claims, links, citations, tasks, tags, ontology nodes, embeddings, etc.
- **Layer 3:** a **provenance/derivation layer** that records *why* a fact exists, *which document states support it*, and *which branch or replica introduced it*. Provenance is a first-class requirement in collaborative KGs, not a nice extra.

At that point, the state of the art is **not** one giant CRDT for everything. It is usually a **coupled architecture**:

1. a **document CRDT / change DAG** for the editable surface, and
2. a **graph-structured knowledge model** with stable IDs, provenance, and branch-aware views on top.

Automerge-style heads and change graphs are useful for the document/history side, while graph CRDT work shows how graph data itself can be replicated and merged.

The biggest shift is this:

A document is no longer just “content.” It becomes a **source of assertions**.

So instead of storing only “paragraph 7 changed,” you also store things like:

- entity `Engine:N4.50`
- claim `has_sender_thread = M10x1`
- support = span(s) in document versions A, B, C
- confidence / validation status
- branch visibility = `research/repower`, `install/final`, etc.

That means your merge semantics must distinguish between at least **three different things**:

- **surface edits**: text/block movement, formatting, deletion
- **semantic assertions**: facts, links, classifications
- **derived knowledge**: extracted triples, summaries, backlinks, embeddings, inferred relationships

Those do **not** all want the same merge rule. Plain CRDT convergence is enough for surface edits, but it is often *insufficient* for semantic truth. A merged system can converge mechanically while still producing a nonsensical or contradictory knowledge graph. Work on local-first correctness and invariants exists precisely because mergeable replicated types do not automatically preserve application-level meaning.

So the modern answer is:

## 1) Give every semantic object a stable identity

Do **not** key knowledge by text position alone.

You want stable IDs for:

- documents
- blocks / nodes
- entities
- claims
- citations / evidence records
- relations
- extraction runs / agent actions

Text anchors can drift; semantic IDs should not. This is the same reason rich-text systems moved beyond naive positional models for overlapping annotations. Peritext shows that naive text-based approaches fail badly once formatting and overlapping structure matter.

## 2) Treat provenance as part of the data model

In a distributed knowledge base, “what do we know?” is incomplete without “according to what?” and “as of which branch/version?” Provenance frameworks for KGs emphasize that provenance is critical for trust, reproducibility, and updateability. Wikidata-style collaborative KGs also rely heavily on references for exactly this reason.

In practice, each fact should be able to point to:

- supporting document spans
- the document version frontier or heads
- extractor / human author / agent identity
- timestamp / logical clock
- confidence / status
- supersedes / disputes / invalidates relations

Without that, overlapping branches will quickly create “ghost facts” that look globally true but were only true in one branch or one interpretation.

## 3) Separate **asserted facts** from **derived facts**

This is the biggest design mistake people make.

A user assertion like:

- “N4.50 panel cutout is 220×160 mm”

is not the same thing as:

- “an extractor inferred a cutout dimension from a PDF diagram”
- “an LLM guessed the cutout from image OCR”
- “a workflow derived a compatibility relation from multiple claims”

Derived-information frameworks for dynamic KGs explicitly model how one piece of information is obtained from others; you want that same idea here.

So your graph should distinguish:

- **asserted**
- **extracted**
- **inferred**
- **computed**
- **deprecated**
- **contested**

## 4) Make branches first-class in the knowledge layer too

This is the crucial part for your original question.

If the document layer supports many heads/branches, but the KG is “global latest only,” you lose most of the value. Automerge’s model already exposes multiple heads in the change graph; your semantic layer should support the same idea conceptually.

You want queries like:

- “show facts true on branch `engine-repower-beta60`”
- “show claims common to these 4 branches”
- “show assertions introduced by branch X but unsupported on main”
- “show all entities whose evidence spans were deleted in this merge”
- “show conflicting values for property P across active branch frontiers”

That means your KG is not just a graph of facts. It is a graph of facts **parameterized by version frontier**.

## 5) Use CRDTs for structure, but constraints for meaning

Graph CRDTs are promising for replication and convergence, but real knowledge bases also need semantic invariants:

- unique canonical entity resolution
- type constraints
- cardinality limits
- ontology/domain/range rules
- no dangling evidence references
- conflict policies for contradictory claims

Graph-CRDT research and local-first safety work both point in this direction: convergence is necessary, but not sufficient, so applications need invariant enforcement and policy above the replicated substrate.

So the actual architecture should be:

- **replicated op log / causal DAG**
- **CRDT materialized views** for doc/tree/text state
- **graph store** for entities/claims/provenance
- **constraint engine** for semantic validity
- **branch-aware query engine**
- optional **derived views / indexes / embeddings**

## 6) Prefer “knowledge attached to document atoms,” not free-floating extraction only

If the KB is layered over a document system with lots of overlapping edits, the most robust design is to anchor knowledge to **block/node/span identities** in the document model whenever possible.

For example:

- block node ID `b:123` contains engine spec table
- claim `c:889` extracted from cells in `b:123`
- evidence points to cell/node IDs, not only character offsets
- if the block moves in a tree CRDT, evidence still follows it

This is where move-aware tree CRDTs matter. If your document is block/tree structured, concurrent moves are not an edge case; they are normal. Loro’s movable tree and the replicated-tree move work are directly relevant here.

## 7) Make “conflict” explicit at the semantic level

With a distributed KB, you should stop pretending every merge resolves to one truth automatically.

Two branches may legitimately produce:

- `engine_power_hp = 50`
- `engine_power_hp = 60`

Both can coexist if they are:

- from different candidate configurations,
- from different evidence sources,
- or from different branch contexts.

So instead of overwriting, represent:

- **parallel claims**
- conflict relation
- branch scope
- adjudication status
- chosen canonical value for a given view

That is much closer to how collaborative KGs and patch-based systems behave in practice than a naive last-writer-wins fact table.

## Practical design recommendation

If I were designing this today, I would use:

**Document plane**

- CRDT-backed block/tree document model
- Peritext-like semantics for rich text/annotations
- move-aware tree support for blocks/subtrees

**Knowledge plane**

- entity/claim/evidence/provenance graph
- every node/edge has stable IDs
- support for branch/frontier scoping
- explicit contradiction/supersession links

**History plane**

- causal op DAG
- named branches as views over frontiers
- time-travel queries over both docs and facts

**Policy plane**

- constraints and invariant checks
- extraction pipelines that emit derived facts, never silently overwrite asserted facts
- human review / adjudication workflows for semantic conflicts

## Bottom line

Once you add a structured, distributed knowledge base, the state of the art becomes:

**branch-aware documents + branch-aware provenance-rich knowledge graphs + explicit semantic conflict management.**

The hard problem is no longer “how do I merge edits?”
It becomes:

**“how do I preserve causality, evidence, identity, and truth-status across many overlapping branches of both content and knowledge?”**

That is a substantially harder system than collaborative text editing, and it is why the winning architecture is usually **document CRDTs underneath, provenance-rich KG above, and invariant/policy logic on top**, rather than one universal merge engine for everything.

The core shift is this:

A “comprehensive, collaborative, trust-based KB” is not mainly about storing triples or documents. It is about storing **claims, provenance, trust, scope, and adjudication** as first-class objects. Collaborative KGs like Wikidata already rely heavily on references/provenance because trust in a shared graph depends on being able to inspect where a statement came from, not just what the statement is.

So the primary unit should usually be **not**:

- entity → property → value

but rather:

- **claim**: “X has property P with value V”
- **evidence**: source spans, documents, measurements, imports
- **provenance**: who/what asserted it, when, from which branch/frontier
- **trust signals**: author reputation, source quality, corroboration, freshness
- **status**: proposed / accepted / contested / superseded / deprecated
- **scope**: globally accepted, branch-local, team-local, confidential, hypothetical

That is much closer to how real collaborative KGs have to work when quality matters. Provenance and references are central because they let users evaluate relevance and authoritativeness of claims.

## What “trust-based” really implies

A trust-based KB should not collapse trust into a single score. In practice, trust is usually **multi-dimensional**:

- **source trust**: is the cited source authoritative?
- **contributor trust**: has this actor been reliable in this domain?
- **corroboration trust**: do multiple independent sources support the claim?
- **procedural trust**: was the claim machine-extracted, human-reviewed, or formally verified?
- **contextual trust**: trusted for which audience, branch, or policy domain?

Recent work on trust and accountability in KG-based AI emphasizes that trust comes from context, semantics, provenance, and accountability structures, not just raw graph connectivity.

So instead of storing:

`engine_power_hp = 50`

you want something like:

`claim_id: c123`
`subject: engine:N4.50`
`predicate: has_power_hp`
`object: 50`
`asserted_by: user:jeremy`
`supported_by: source:s456, source:s812`
`trust_vector: {source_authority: high, corroboration: medium, human_reviewed: true}`
`status: accepted_on_main, contested_on_branch_beta60`
`valid_for_frontier: F_main_2026_03_06`

That is the right granularity.

## “Comprehensive” changes the design again

A comprehensive KB means you will have **multiple knowledge regimes mixed together**:

- hard facts
- soft hypotheses
- procedures
- plans
- derived summaries
- extracted entities
- inferred relations
- contradictory alternatives
- private notes
- branch-specific design options

Those should not all live in one undifferentiated truth table. A comprehensive system needs to distinguish at least:

- **asserted facts**
- **observations**
- **measurements**
- **citations**
- **inferences**
- **recommendations**
- **alternatives**
- **decisions**
- **tasks / intentions**

Otherwise the system will blur “what is true” with “what was suggested” and “what was inferred.” That distinction is one of the core issues in provenance and traceability work for KGs, especially when users need to audit how knowledge was formed over time.

## Distributed + collaborative means branch-aware truth

Once the system is distributed and collaborative, truth is no longer just global current state. You need to represent:

- what is accepted on `main`
- what is proposed on `research/foo`
- what is disputed between branches
- what is visible only to a trust circle
- what is common across several frontiers

That means your KB should be **branch-aware** the same way modern local-first document systems are version-aware. Local-first systems use replicated data types/CRDTs to guarantee convergence, but the current research is very clear that convergence alone is not enough; application invariants and correctness constraints must also be enforced above the merge substrate.

So the right model is:

- **replication layer** for availability and offline work
- **knowledge layer** for claims/evidence/provenance
- **policy layer** for trust, invariants, and adjudication

Not one giant magical merge engine.

## The right architecture

I would break it into five planes.

### 1. Replication plane

Use a causal op-log / CRDT-style substrate for distributed updates and eventual convergence. Graph-specific replicated data types exist, and there is active work on graph CRDTs and replicated graph databases.

### 2. Identity plane

Every important object gets a stable ID:

- entity
- claim
- evidence record
- source artifact
- branch
- frontier
- decision
- user / agent / service

Stable identity matters because text positions, extracted labels, and even graph structure can drift as collaboration continues.

### 3. Epistemic plane

Represent knowledge as **claims with provenance and status**, not only facts. This is where you store:

- support / refute edges
- supersedes / duplicates
- confidence
- trust vector
- acceptance state
- branch scope
- temporal validity

### 4. Governance plane

A trust-based system needs formal governance:

- who may assert vs approve
- what sources are admissible
- how conflicts are resolved
- what invariants are required
- what is machine-only vs human-certified

This is where “trust” becomes operational instead of decorative.

### 5. Query plane

Queries should be able to ask:

- “what is currently accepted?”
- “what is accepted by highly trusted sources only?”
- “what changed in accepted knowledge between branch A and B?”
- “show unsupported claims”
- “show claims derived only by automation”
- “show contradictions with no adjudication”
- “show all facts whose provenance chain includes this source”

That is far more powerful than ordinary KG lookup.

## What trust computation should look like

I would not compute one monolithic truth score. I would compute a **trust envelope**.

For each claim, derive something like:

- **support count**
- **independence score** of supporting sources
- **source authority score**
- **editor reliability score**
- **freshness decay**
- **review state**
- **conflict pressure**
- **policy compliance**

Then expose policy-defined views such as:

- **strict view**: only human-reviewed claims with authoritative sources
- **exploratory view**: includes machine-extracted and weakly supported claims
- **team view**: trust claims from specific collaborators more highly
- **compliance view**: only claims with complete provenance chains

That matches the idea that trust is contextual and view-dependent, which is a recurring theme in work on collaborative KGs and trustworthy KG-based AI.

## The biggest technical danger

The biggest failure mode is building a system that is **distributed at the storage layer but centralized in epistemology**.

Examples:

- every merge collapses to one “latest truth”
- machine-extracted claims overwrite curated claims
- provenance is optional metadata instead of a required edge
- branch-local hypotheses leak into accepted global knowledge
- trust is stored as an opaque number no one can audit

That kind of system will converge mechanically and still become untrustworthy.

The local-first literature is useful here: convergence does not guarantee correctness, and invariants have to be preserved explicitly.

## The actual state of the art, conceptually

The state of the art for the thing you are describing is roughly:

- **replicated/local-first substrate** for distributed collaboration,
- **provenance-rich collaborative KG** for knowledge representation,
- **policy/invariant engine** for correctness,
- **trust and accountability layer** for acceptance and explanation,
- **branch-aware views** instead of one flat canonical graph.

If you want one sentence:

**A serious trust-based distributed KB should behave more like a versioned, policy-governed system of claims and evidence than like a database of facts.**

## 1. The core thesis

It means nothing essential may require surrender.

- **Autonomy Core** defines shared constitutional protocol law.

## 2. The sovereignty line

**Above the sovereignty line** sit the things the operator must control:
identity, keys, policy, module activation, trust settings, agent permissions, local overrides, UI composition, portability, and the ability to revoke delegation.

**Below the sovereignty line** sit things that may be delegated:
relay, storage, search, indexing, model inference, media processing, GIS overlays, market feeds, archives, analytics, and high-availability coordination services.

That means Autonomy Infra can be centralized.
It just cannot be sovereign.

The constitutional layer published on the root domain. It defines the universal primitives, discovery surfaces, compatibility rules, event envelopes, capability model, and policy hooks.

Domain protocol packages. `/real-estate` is not an endpoint bucket. It is a typed domain law: objects, relations, events, capabilities, trust rules, invariants, and extension points.

The user-facing sheet of glass. It is a projection of the runtime, not the owner of it.

### Autonomy Infra

Optional providers of compute, storage, transport, search, AI, data feeds, and other acceleration services. They serve the runtime. They do not own the network.
